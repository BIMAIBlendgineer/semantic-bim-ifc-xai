# Validation Gates Specification

## Overview

Every record proposed for ingestion into the semantic BIM/IFC dataset must pass through an 11-stage validation matrix before it is assigned to any dataset split. The gate sequence enforces a fail-closed policy: any stage failure immediately blocks the candidate from progression.

---

## Gate Matrix

### Gate 1 — JSON Parse Validation

**What it checks**: The record can be parsed as syntactically valid JSON.

**Failure condition**: Any JSON syntax error (unterminated strings, missing brackets, trailing commas).

**Rationale**: Downstream schema validation and model training require syntactically correct inputs. A malformed record would silently corrupt batch processing pipelines.

---

### Gate 2 — Schema Validation

**What it checks**: The record conforms to the Pydantic-defined semantic record contract schema.

**Failure condition**: Missing required fields, incorrect data types, or values outside allowed enumerations.

**Rationale**: The schema encodes domain knowledge. Records that omit required fields (such as `evidence_trace` or `suggested_ifc_class`) cannot train the model to produce those fields.

---

### Gate 3 — IFC Class Candidate Validation

**What it checks**: The `suggested_ifc_class` field references a valid entity from the buildingSMART IFC specification.

**Failure condition**: A class name not in the official IFC registry (e.g., fabricated names like `IfcGenericElement`).

**Rationale**: Models trained on invalid IFC classes will hallucinate non-existent schema entities, breaking downstream BIM software integration.

---

### Gate 4 — LOI/LOD Consistency

**What it checks**: The Level of Information (LOI) properties listed in `loi_table` are consistent with the suggested IFC entity type and standard IFC property sets (Psets).

**Failure condition**: Properties that do not belong to any standard Pset associated with the declared IFC class.

**Rationale**: Mismatched LOI properties indicate a semantic grounding failure — the model has invented properties that do not exist for that element type.

---

### Gate 5 — Evidence Trace Completeness

**What it checks**: The `evidence_trace` field contains at least one reference to an entity, property, relationship, or rule that can be traced back to the input context.

**Failure condition**: Empty evidence trace, or evidence referencing entities not present in the input.

**Rationale**: Evidence completeness is the core XAI requirement. A record without a valid evidence trace cannot train the model to explain its decisions.

---

### Gate 6 — Hallucination / Unsupported Claim Scan

**What it checks**: Automated comparison of all generated properties and class claims against the input context and canonical catalogues.

**Failure condition**: Properties or classes generated by the model that are absent from both the input context and the canonical catalogue.

**Rationale**: Hallucinated claims corrupt the training signal and create safety risks in engineering contexts where incorrect classification could affect structural safety.

---

### Gate 7 — Replay Validation

**What it checks**: Deterministic re-execution of the full pipeline on the input prompt produces a structurally consistent and schema-valid output.

**Failure condition**: Non-deterministic output, schema invalidity on replay, or intent mismatch.

**Rationale**: Reproducibility is a scientific requirement. Records that cannot be consistently replayed cannot serve as reliable benchmark test cases.

---

### Gate 8 — Leakage and Deduplication Check

**What it checks**: String hashing and semantic similarity filtering across all dataset splits to detect near-duplicate records and potential train/test contamination.

**Failure condition**: Cosine similarity above threshold with any existing record in a different split.

**Rationale**: Train/test contamination inflates benchmark scores and produces misleading performance estimates.

---

### Gate 9 — Sanitization Scan

**What it checks**: Automated regex and Named Entity Recognition (NER) scan to detect and redact private project names, geographic locations, IP addresses, local folder paths, and person names.

**Failure condition**: Unredacted private identifiers detected in any field.

**Rationale**: Public datasets must not expose real-world project data, client identifiers, or infrastructure details.

---

### Gate 10 — Forbidden Patterns Scan

**What it checks**: File-level scan for prohibited strings: real API keys, real access tokens, Windows absolute working paths, private model filenames, and product-specific failure codes.

**Failure condition**: Match of a real credential pattern (e.g., actual `sk-` prefixed API key with high-entropy value), real Hugging Face token, or internal absolute path.

> [!NOTE]
> Academic explanatory text discussing credential patterns as examples (e.g., in policy documentation) is permitted. Only real credential values are prohibited.

**Rationale**: Credentials committed to a public repository represent an immediate security incident.

---

### Gate 11 — Public/Private Boundary Verification

**What it checks**: No record or file marked for public release contains references to private infrastructure (internal database schemas, internal dataset splits, private model checkpoints, commercial tenant identifiers).

**Failure condition**: Any field referencing a private asset that has not been sanitized or marked as excluded.

**Rationale**: The public research surface must be clearly separated from the private production and training infrastructure.

---

## Gate Decision Outcomes

| Outcome | Meaning |
|---|---|
| `pass` | Record cleared all 11 gates; eligible for split assignment |
| `blocked` | Record failed one or more gates; must be corrected and re-submitted |
| `rejected` | Record failed sanitization or hallucination scan; removed from dataset pipeline |
| `eval_only` | Record is high-complexity or edge-case; restricted to evaluation use only, not training |

---

## Relationship to XAI

Gates 5 (Evidence Trace) and 6 (Hallucination Scan) directly enforce the XAI requirement at the dataset construction level. A model trained on records that passed these gates is exposed, from the start, to the requirement that every classification must be grounded in explicit evidence. This upstream enforcement is the foundation of the XAI evaluation methodology documented in [xai_evaluation_position.md](xai_evaluation_position.md).
